Rendering is the single largest lever this project has on performance and SEO. Before any component or route is written, the engineering documentation asks where its HTML gets produced and when — on the server ahead of time, on the server per request, or in the browser after load. `ADR-0006` locks the answer for this codebase: a server-first, static-biased posture, with streaming reserved for the pages that genuinely benefit from it. This article defines the four standard rendering strategies and then explains, concretely, which of them this site actually uses and why.

## The four strategies

### SSR (Server-Side Rendering)

SSR renders a page's HTML on the server at request time, for every request. It guarantees fresh, request-aware output and gives crawlers complete markup without executing JavaScript, but it adds per-request compute and latency, and it forfeits the cheap edge caching that build-time HTML gets for free. `ADR-0006` rejects SSR-on-every-request as the _default_ for this project: with content fixed at build time, paying for per-request rendering would add cost with no benefit.

### CSR (Client-Side Rendering)

CSR ships a mostly empty HTML shell and lets the browser fetch data and render the page with JavaScript after load. It can feel fluid for highly interactive, app-like views, but it means a large client bundle, hydration of the entire tree, and markup that is invisible to simple crawlers until scripts execute — a direct hit to Core Web Vitals and SEO. `ADR-0001` and `ADR-0006` both name a client-first SPA as the anti-pattern this architecture is built to avoid, since it conflicts with the product principle of _performance over visual effects_.

### SSG (Static Site Generation)

SSG produces a page's HTML at build time, once, and serves the same static output to every visitor from the edge. It is the cheapest and fastest option whenever the content is known ahead of time, because there is nothing left to compute per request — the trade-off is that the content can only change on a new build. Because this project's content — projects, case studies, articles, journey entries — lives in the repository as MDX and structured data with no runtime database, SSG is the natural fit and the strategy `ADR-0006` adopts as the default for nearly every route.

### ISR (Incremental Static Regeneration)

ISR extends static generation by letting a statically built page be regenerated in the background after a configured interval or on demand, without a full redeploy. It suits content that changes independently of the codebase, such as a CMS-backed page, at the cost of added caching complexity and the possibility of briefly stale output. This project deliberately does **not** use ISR: since a content change _is_ a deploy, a rebuild already regenerates every affected page, so reaching for `revalidate` would add complexity for no benefit. ISR is reserved for a future content source that can change without a deploy, out of scope for V1.

## Server Components first

The App Router's default execution context is the React Server Component, and this codebase treats that default as non-negotiable: a component is a Server Component unless it must run on the client, and you do not justify writing one — you justify not writing one. `"use client"` is the deliberate exception, valid only for genuine interactivity such as event handlers, `useState`/`useEffect`, browser-only APIs like `window` or `localStorage`, client Context, or a stateful third-party widget. Marking a directive is not enough on its own — the boundary must sit at the smallest possible leaf, since `"use client"` pulls its entire import subtree into the client bundle. A Server Component can render a Client Component, and server-rendered content can be passed into a client shell as `children` so it never joins the client bundle.

## How this site renders

Because all content is known at build time, routes are statically generated by default and regenerated on every deploy. Dynamic route segments — a project slug, a locale — are enumerated ahead of time with `generateStaticParams`, so `app/[locale]/projects/[slug]/page.tsx` pre-renders one static page per locale and per slug rather than rendering on demand. Closed sets of params keep `dynamicParams` set to `false`, so an unrecognized slug resolves to a genuine not-found rather than triggering an on-demand render. A route only becomes dynamic when it depends on real request-time input that cannot be enumerated — an unbounded `searchParams` space, or reading `cookies()`/`headers()` — and the one routine exception in this system is the contact form's server action, whose POST is inherently dynamic without making the surrounding page dynamic.

## Streaming

For a fully static page there is little to stream, since the HTML is already complete by the time it is served. Streaming earns its place on pages with a genuinely slower or independently-resolving subtree: a `<Suspense>` boundary wraps that subtree specifically, with a fallback that mirrors the final layout, so the fast shell paints immediately while the slower part resolves afterward. A `loading.tsx` file works the same way at the segment level, acting as an implicit Suspense boundary around a route's `page.tsx`. Boundaries are placed deliberately around genuinely slow parts, not scattered reflexively across every component.
